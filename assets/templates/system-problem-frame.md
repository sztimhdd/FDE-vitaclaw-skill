# System Problem Frame

## Use

Use after reality capture to define the actual problem boundary and bottleneck.

## Required Fields

- Problem class
- System boundary
- Primary bottleneck
- North-star outcome
- Non-goals
- Critical assumptions

## Optional Fields

- Competing stakeholder asks
- Deferred subproblems

## Typical Anti-Pattern

The frame is a feature wishlist instead of a bottleneck definition.

## Completion Standard

The team can say what is being solved now, what is explicitly deferred, and what assumption could invalidate the plan.
